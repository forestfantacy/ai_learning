# AI 数据分析 SOP

这个 skill 用来把“分析一下这份数据”这类模糊请求，推进成面向决策、有证据链、可验证、可行动的业务数据分析。适用对象包括表格数据、Excel/BI 数据、SQL 导出、CSV/XLSX 文件、Notebook、仪表盘和分析报告草稿。

## 核心规则

不要从一个可见变化直接跳到原因。始终把下面六件事拆开：

1. **决策目标**：谁要基于分析做什么判断。
2. **指标契约**：定义、过滤条件、粒度、时间窗口、数据源和口径风险。
3. **数据事实**：从数据中实际计算或观察到了什么。
4. **业务解释**：为什么可能发生，这一步必须标注推断和假设。
5. **证据强度**：置信度、替代解释、缺失数据和下一步验证。
6. **行动闭环**：负责人、建议动作、预期效果、验证指标、风险和复盘时间。

## 工作流

1. **接收与盘点**
   - 先检查可用文件或数据表、字段结构、行数、时间范围、主键/外键和缺失情况。
   - 如果用户没有明确决策目标，先做目标发现，再进入计算。
   - 如果没有数据，不要编造结论，输出分析方案和数据需求清单。
   - 如果缺失信息会阻塞下一步分析，按交互协议向用户索取最少必要信息。

2. **目标发现**
   - 基于字段、业务对象、指标类型和潜在使用者，生成候选分析目标。
   - 按业务价值、数据支持度、可行动性、异常信号和决策对象相关性排序。
   - 只有当多个目标会导致明显不同的分析路径时，才向用户确认优先方向。

3. **指标契约**
   - 明确业务对象、数据粒度、分子、分母、时间窗口、时间口径、过滤条件、权威数据源和口径风险。
   - 对“销售额”“活跃用户”“转化率”“利润”等模糊指标，不要默认口径正确；必须先显式说明假设或要求确认。

4. **业务模型还原**
   - 识别每张表的粒度和表间关联。
   - 将字段划分为结果指标、过程指标、基础字段和分析维度。
   - 建立分解公式，例如：销售额 = 订单量 * 客单价，毛利 = 收入 - 成本，退货率 = 退货订单数 / 总订单数。

5. **分析执行**
   - 有数据文件时，优先使用可复现代码或结构化查询，不只靠手工推理。
   - 在报告中分开说明数据提取、清洗处理、指标计算和业务解释。
   - 归因前必须完成异常检查、指标分解、分层对比和质量闸门。

6. **归因分析**
   - 先定位损失或增长发生在哪里，再生成候选原因。
   - 用时间一致性、对象特异性、对照组和内部数据解释力验证候选原因。
   - 对场外因素必须标注为假设，除非已有外部数据或对照证据支持。

7. **输出交付**
   - 每条关键结论都要转换为：数据事实、业务判断、建议动作、验证指标、风险和下一步数据需求。
   - 每个重要判断都要给出置信度、替代解释和缺失证据。

## 脚本

使用 `scripts/create_analysis_brief.py` 生成 Markdown 分析报告骨架：

```bash
python3 ai-data-analysis-plus/scripts/create_analysis_brief.py \
  --title "销售额下降诊断" \
  --stakeholder "运营负责人" \
  --decision "判断优先介入哪个渠道" \
  --output analysis_brief.md
```

脚本只生成结构骨架。最终交付时必须填入真实计算结果和证据，不要保留占位式结论。

## 交付标准

一份完整回答通常应包含：

- 决策目标和关键假设。
- 指标契约。
- 数据盘点和质量说明。
- 候选分析目标或已选择的分析路径。
- 关键计算过程和核心发现。
- 归因证据表。
- 置信度、替代解释和缺失数据。
- 包含负责人、验证指标、风险和复盘时间的行动方案。
