# 交互协议

本协议用于多轮数据分析：当当前信息不足以继续分析时，明确告诉用户已经分析到哪一步、为什么卡住、最少还需要什么数据；用户补充后，从卡住的位置继续推进，不要从头重来。

## 什么时候必须追问

出现以下情况时，必须暂停关键结论或归因，不要继续猜测：

| 场景 | 处理方式 |
|-|-|
| 决策目标不清，且不同目标会导致不同分析路径 | 给出候选目标并请用户选择优先方向。 |
| 指标口径不清，且口径会改变结论 | 说明默认假设，并询问用户确认或补充口径。 |
| 缺少关键数据，导致无法验证归因机制 | 输出当前证据、置信度和最少数据需求。 |
| 样本量太小或时间窗口太短 | 降低置信度，并请求更长时间或更多样本。 |
| 表间关联键缺失或不唯一 | 请求主键/外键说明，或要求提供映射表。 |
| 数据质量问题可能影响结论 | 先报告质量风险，再请求修正数据或确认处理规则。 |

## 提问原则

- 一次只问会影响下一步分析的关键问题，通常控制在 1-3 个。
- 优先索取“最少可继续分析”的数据，不要一口气要完整理想数据集。
- 问题必须说明用途：为什么需要这项数据，它会验证哪个假设。
- 如果可以在多个数据中选择，按优先级列出“最关键、其次、可选”。
- 对用户已经提供过的信息，不要重复追问；直接纳入当前证据链。

## 阻塞时的输出格式

当缺失数据阻塞下一步归因或结论时，使用这个结构：

```markdown
当前已经确认：
- <事实 1>
- <事实 2>

当前置信度：<高/中/低>，因为 <原因>。

现在不能继续确认的是：
- <待验证的机制或原因>

继续分析最少需要：
| 优先级 | 需要的数据 | 用途 | 可接受形式 |
|-|-|-|-|
| 必需 | <字段/表/截图/口径说明> | <验证哪个假设> | <CSV/Excel/文字说明/截图> |
| 可选 | <字段/表> | <提高置信度或做分层> | <形式> |

你补充后，我会继续：
1. <下一步分析动作>
2. <会更新的结论或置信度>
```

## 目标不清时的输出格式

```markdown
这份数据可能支持几个不同方向，建议先选一个主目标：

| 选项 | 分析目标 | 能回答的问题 | 需要的数据支持 | 适合的决策 |
|-|-|-|-|-|

我建议优先做：<目标>，原因是 <业务价值/数据支持/可行动性>。
```

如果用户没有选择，但可以根据上下文合理推进，选择最高优先级目标，并显式说明假设。

## 口径不清时的输出格式

```markdown
继续计算前需要确认指标口径：

| 指标 | 当前默认假设 | 为什么会影响结论 |
|-|-|-|

如果你没有特别口径，我将按上述假设继续，并在报告中标注为口径假设。
```

当用户未回复且任务可以低风险推进时，可以按默认假设继续；当口径会造成相反结论时，必须等待确认。

## 用户补充数据后的恢复规则

用户补充新数据或口径后：

1. 先简短复述收到的补充内容。
2. 更新数据盘点和指标契约。
3. 从上一次阻塞点继续，不要重做无关分析。
4. 重新计算受影响指标。
5. 更新归因证据表和置信度。
6. 明确说明哪些旧结论被保留、修正或撤回。

恢复输出格式：

```markdown
已收到补充：<数据/口径说明>。

我将从 <上次阻塞点> 继续，而不是重做全部分析。

本次更新影响：
- 保留：<仍成立的结论>
- 修正：<需要调整的结论>
- 新增：<新增发现>

更新后的置信度：<高/中/低>。
```

## 置信度更新规则

| 置信度 | 条件 |
|-|-|
| 高 | 数据覆盖关键口径、样本足够、机制清楚、有对照或稳定性验证。 |
| 中 | 现象和位置清楚，但机制缺少一到两个关键验证。 |
| 低 | 只有表面现象、样本不足、口径不清或替代解释同样成立。 |

补充数据后，不要默认提高置信度。只有当新数据实际验证了机制、排除了替代解释或提高了样本可靠性时，才上调。

## 不要做的事

- 不要因为缺数据就编造外部原因。
- 不要把“损失位置”写成“根因”。
- 不要一次索要所有可能有用的数据。
- 不要在用户补充数据后从头输出一份冗长报告，除非用户要求。
- 不要隐藏假设；默认假设必须写入报告。
