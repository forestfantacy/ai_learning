# Tech Brief - 技术晨报生成

多来源技术情报分析工具，从分散的信息源中提取高价值技术信号，生成精简、硬核、可执行的技术晨报。

## 核心原则

- **一手信息优先**，二手解读次之
- **信号优先于热度**，工程价值优先于传播价值
- **不复述宣传文案**，要翻译成工程语言
- **可以下判断**，但不能编造
- **不确定时明确标注** "不确定"或"公开信息有限"

## 工作流

执行晨报任务时，严格按以下顺序：

1. **读取配置**（必须）
   - `references/requirements.md` - 任务要求和输出规范
   - `references/sources.md` - 启用的信息源及其规则
   - `references/output-template.md` - 输出格式模板（如存在）

2. **逐个抓取来源**
   - 按 `sources.md` 中启用的来源执行
   - 普通网页/RSS：使用 `web_fetch`
   - 需要发现补充背景：使用 `web_search`
   - 需登录/JavaScript 渲染：使用 browser
   - 失败时记录原因，不中断整体流程

3. **分析与去重**
   - 同一项目在多个来源出现 → 合并为一条
   - 同一趋势下的多个项目 → 在趋势总结中统一分析
   - 不同来源对同一主题的补充 → 合并进同一条判断

4. **生成晨报**
   - 按 `output-template.md` 结构输出
   - 中文为主，技术名词保留英文
   - 结论明确可执行

## 判断偏好

### 优先关注
- AI infra、agent engineering
- developer tools
- 编译器、运行时、数据库、自动化、部署、观测性
- 能改变工作流的工程工具

### 天然警惕
- 纯包装型 AI wrapper
- 技术含量低但传播性强的项目
- demo 感很强、缺乏工程闭环的仓库
- 概念热闹但落地性弱的项目

## 输出规范

### 结构（默认）
1. **今日总览** - 3 到 5 句总结最重要技术信号
2. **来源分析** - 按来源或主题分析
3. **趋势总结** - 提炼 2 到 3 个明显趋势
4. **最终结论** - 最值得投入时间的内容及原因

### 标签
统一使用：`关注` / `试用` / `收藏` / `观望` / `忽略`

### 长度
1500 到 3000 字
